content: "# TLDR\n\nI have a use case for nesting directories with cascading .envrc
  files. and ran\ninto an issue where the parent environment was being reset by the
  child\nenvironment. The fix is `source_up`\n\n# Context\n\nI manage some infrastructure
  at home with a folder like this:\n\n```bash\n.\n\u251C\u2500\u2500 homelab\n\u2502
  \  \u251C\u2500\u2500 .envrc\n\u2502   \u2514\u2500\u2500 stack1\n\u2502       \u251C\u2500\u2500
  .envrc\n```\n\nI do this so that the \"homelab-wide\" terraform variables can be
  exported with\n`TF_VAR_the_variable` in the `homelab/.envrc` and then the stack
  specific\nvariables in `stack1/.envrc` get exported like `TF_VAR_stack_specific_var`.\n\nFor
  example, my cloudflare account id is needed in every stack, but the\ncloudflare
  zone id is different per stack - so instead of copying the account\nid var to every
  stack (however the vars are managed), I can set it in a parent\n.envrc and share
  it in nested stack directories\n\n## Problem\n\n```\n# homelab/.envrc\n#!/bin/bash\nexport
  VAR_FOO=BAR\n```\n\nSo when I `cd homelab` I get\n\n```\nnic in /tmp/homelab  \uF427
  (dev) \U000F0484 \U000F150E NO PYTHON ENVIORNMENT SET\n\u2B22 [devbox] \u276F direnv
  allow\ndirenv: loading /tmp/homelab/.envrc\ndirenv: export +VAR_FOO\n\nnic in /tmp/homelab
  \ \uF427 (dev) \U000F0484 \U000F150E NO PYTHON ENVIORNMENT SET\n\u2B22 [devbox]
  \u276F env | grep VAR\nVAR_FOO=BAR\n\n```\n\nYou can see `VAR_FOO` is set... but
  now I'll `cd stack1`\n\n```\nnic in /tmp/homelab  \uF427 (dev) \U000F0484 \U000F150E
  NO PYTHON ENVIORNMENT SET\n\u2B22 [devbox] \u276F cd stack1\ndirenv: error /tmp/homelab/stack1/.envrc
  is blocked. Run `direnv allow` to approve its content\n\nnic in /tmp/homelab/stack1
  \ \uF427 (dev) \U000F0484 \U000F150E NO PYTHON ENVIORNMENT SET\n\u2B22 [devbox]
  \u276F direnv allow\ndirenv: loading /tmp/homelab/stack1/.envrc\ndirenv: export
  +VAR_BAZ\n\nnic in /tmp/homelab/stack1  \uF427 (dev) \U000F0484 \U000F150E NO PYTHON
  ENVIORNMENT SET\n\u2B22 [devbox] \u276F env | grep VAR\nVAR_BAZ=BAM\n\n```\n\nAs
  you can see `VAR_FOO` was unset, and `VAR_BAZ` is set... so `homelab/.envrc` was
  blown away!\n\n## Fix\n\nEdit `stack1/.envrc` with a `source_up` command\n\n```\n#
  stack1/.envrc # NEW\n#!/bin/bash\nsource_up  # source the parent environment\nexport
  VAR_BAZ=BAM\n\n```\n\nAfter the edit\n\n```\nnic in /tmp/homelab/stack1  \uF427
  (dev) \U000F0484 \U000F150E NO PYTHON ENVIORNMENT SET\n\u2B22 [devbox] \u276F vim
  .envrc\ndirenv: error /tmp/homelab/stack1/.envrc is blocked. Run `direnv allow`
  to approve its content\n\nnic in /tmp/homelab/stack1  \uF427 (dev) \U000F0484 \U000F150E
  NO PYTHON ENVIORNMENT SET  took 5s\n\u2B22 [devbox] \u276F direnv allow\ndirenv:
  loading /tmp/homelab/stack1/.envrc\ndirenv: loading /tmp/homelab/.envrc\ndirenv:
  export +VAR_BAZ +VAR_FOO\n\nnic in /tmp/homelab/stack1  \uF427 (dev) \U000F0484
  \U000F150E NO PYTHON ENVIORNMENT SET\n\u2B22 [devbox] \u276F env | grep VAR\nVAR_FOO=BAR\nVAR_BAZ=BAM\n\n```\n\nBoth
  vars are set!"

# TLDR

I have a use case for nesting directories with cascading .envrc files. and ran
into an issue where the parent environment was being reset by the child
environment. The fix is `source_up`

# Context

I manage some infrastructure at home with a folder like this:

```bash
.
 homelab
    .envrc
    stack1
        .envrc
```

I do this so that the "homelab-wide" terraform variables can be exported with
`TF_VAR_the_variable` in the `homelab/.envrc` and then the stack specific
variables in `stack1/.envrc` get exported like `TF_VAR_stack_specific_var`.

For example, my cloudflare account id is needed in every stack, but the
cloudflare zone id is different per stack - so instead of copying the account
id var to every stack (however the vars are managed), I can set it in a parent
.envrc and share it in nested stack directories

## Problem

```
# homelab/.envrc
#!/bin/bash
export VAR_FOO=BAR
```

So when I `cd homelab` I get

```
nic in /tmp/homelab   (dev)   NO PYTHON ENVIORNMENT SET
 [devbox]  direnv allow
direnv: loading /tmp/homelab/.envrc
direnv: export +VAR_FOO

nic in /tmp/homelab   (dev)   NO PYTHON ENVIORNMENT SET
 [devbox]  env | grep VAR
VAR_FOO=BAR

```

You can see `VAR_FOO` is set... but now I'll `cd stack1`

```
nic in /tmp/homelab   (dev)   NO PYTHON ENVIORNMENT SET
 [devbox]  cd stack1
direnv: error /tmp/homelab/stack1/.envrc is blocked. Run `direnv allow` to approve its content

nic in /tmp/homelab/stack1   (dev)   NO PYTHON ENVIORNMENT SET
 [devbox]  direnv allow
direnv: loading /tmp/homelab/stack1/.envrc
direnv: export +VAR_BAZ

nic in /tmp/homelab/stack1   (dev)   NO PYTHON ENVIORNMENT SET
 [devbox]  env | grep VAR
VAR_BAZ=BAM

```

As you can see `VAR_FOO` was unset, and `VAR_BAZ` is set... so `homelab/.envrc` was blown away!

## Fix

Edit `stack1/.envrc` with a `source_up` command

```
# stack1/.envrc # NEW
#!/bin/bash
source_up  # source the parent environment
export VAR_BAZ=BAM

```

After the edit

```
nic in /tmp/homelab/stack1   (dev)   NO PYTHON ENVIORNMENT SET
 [devbox]  vim .envrc
direnv: error /tmp/homelab/stack1/.envrc is blocked. Run `direnv allow` to approve its content

nic in /tmp/homelab/stack1   (dev)   NO PYTHON ENVIORNMENT SET  took 5s
 [devbox]  direnv allow
direnv: loading /tmp/homelab/stack1/.envrc
direnv: loading /tmp/homelab/.envrc
direnv: export +VAR_BAZ +VAR_FOO

nic in /tmp/homelab/stack1   (dev)   NO PYTHON ENVIORNMENT SET
 [devbox]  env | grep VAR
VAR_FOO=BAR
VAR_BAZ=BAM

```

Both vars are set!
